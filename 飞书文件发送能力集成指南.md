# 飞书文件发送能力集成指南

本文档用于在本地 OpenClaw 环境中启用、检查并验证飞书文件发送能力。

适用目标：
- Agent 能在飞书会话中正常发送文本消息
- Agent 能把本地文件作为飞书文件消息发送给用户
- 出现异常时，可以按步骤快速定位问题

---

## 一、前置条件

### 1.1 飞书开放平台权限

确保你的飞书应用已经开通并发布以下权限：

- `im:message`：发送消息权限
- `im:message.file`：发送文件消息权限
- `im:resource`：上传资源权限

检查方式：
1. 登录 <https://open.feishu.cn/>
2. 进入对应应用
3. 打开“权限管理”
4. 确认以上权限已添加并完成发布

> 如果权限刚刚新增但没有发布，线上通常不会立即生效。

### 1.2 OpenClaw 已完成飞书通道接入

先确认当前实例已经配置了飞书通道：

```bash
openclaw config get channels.feishu --json
```

期望能看到类似结果：

```json
{
  "provider": "feishu",
  "accountId": "default",
  "appId": "cli_xxx",
  "appSecret": "***",
  "capabilities": ["im", "ccm", "base", "contact", "search", "calendar"]
}
```

如果这里查不到配置，说明飞书通道本身还没有接好，应先完成基础接入。

## 二、配置检查清单

### 2.1 检查飞书通道配置

```bash
openclaw config get channels.feishu --json
```

重点确认：
- `provider` 为 `feishu`
- `appId` 和 `appSecret` 已配置
- 没有明显的空值或拼写错误

### 2.2 检查 Gateway 状态

```bash
openclaw gateway status
```

如果 Gateway 未运行，可使用：

```bash
openclaw gateway start
```

如果刚修改过配置，建议重启：

```bash
openclaw gateway restart
```

### 2.3 检查是否存在工具或能力限制

可先查看是否存在显式 deny 配置：

```bash
openclaw config get tools.deny --json
```

如果输出中包含与消息发送相关的限制项，需要按你的实际配置移除后再测试。

> 注意：不同 OpenClaw 版本、不同插件组织方式下，具体配置键名可能不同。不要机械照抄其他环境中的键名，优先以你本机实际配置为准。

### 2.4 检查飞书相关能力是否已正确加载

如果你使用了额外的飞书技能、扩展或自定义配置，建议同步检查：
- 对应扩展是否已安装
- 配置项是否启用
- 重启 Gateway 后是否生效

如果你不确定配置路径，建议先运行：

```bash
openclaw config get plugins --json
```

确认当前环境里确实已经加载了 Feishu 相关插件或能力。

## 三、推荐自检脚本

将下面内容保存为 `check-feishu-file.sh`：

```bash
#!/usr/bin/env bash
set -e

echo "=== 飞书文件发送能力自检 ==="
echo ""

echo "1. 检查飞书通道配置..."
if FEISHU_CONFIG=$(openclaw config get channels.feishu --json 2>/dev/null); then
  if [ -n "$FEISHU_CONFIG" ] && [ "$FEISHU_CONFIG" != "null" ]; then
    echo "✓ 飞书通道已配置"
  else
    echo "✗ 飞书通道为空，请先完成飞书接入"
    exit 1
  fi
else
  echo "✗ 无法读取飞书通道配置"
  exit 1
fi

echo "2. 检查 Gateway 状态..."
if openclaw gateway status 2>/dev/null; then
  echo "✓ 已执行 Gateway 状态检查"
else
  echo "⚠ Gateway 状态检查失败，请手动执行：openclaw gateway status"
fi

echo "3. 检查 deny 配置..."
if DENY_LIST=$(openclaw config get tools.deny --json 2>/dev/null); then
  echo "当前 tools.deny: ${DENY_LIST}"
else
  echo "⚠ 未读取到 tools.deny，跳过"
fi

echo ""
echo "=== 自检完成 ==="
echo "接下来请在飞书中手动执行一次文本消息测试和文件发送测试。"
```

执行方式：

```bash
chmod +x check-feishu-file.sh
./check-feishu-file.sh
```

## 四、手动验证流程

### 4.1 先验证文本消息

在飞书里对 Agent 发送：

```text
发个测试消息
```

如果文本消息都无法发送，先不要继续排查文件能力，优先修复基础消息通道。

### 4.2 再验证文件消息

在飞书里对 Agent 发送：

```text
把 /绝对路径/某个文件.md 发给我
```

建议优先选一个小于 1 MB、确定存在、纯文本的文件做测试，例如：

```text
把 /home/yourname/.openclaw/workspace/SOUL.md 发给我
```

### 4.3 发送前先确认文件存在

```bash
ls -lh /绝对路径/某个文件.md
```

## 五、常见问题排查

### 5.1 无法发送文件消息

可能原因：
- 飞书应用缺少 `im:message.file`
- 飞书应用缺少 `im:resource`
- 权限已添加但尚未发布
- 文件路径不存在
- 文件不可读
- 文件体积过大

建议排查顺序：
1. 检查飞书权限并确认已发布
2. 检查文件是否存在
3. 检查文件是否可读
4. 用更小的测试文件重试
5. 重启 Gateway 后再次测试

### 5.2 文本能发，文件不能发

这通常说明：
- 基础消息链路正常
- 文件上传或文件消息权限有问题

优先检查：
- `im:message.file`
- `im:resource`
- 飞书开放平台里对应权限是否已发布生效

### 5.3 凭证失效或授权异常

症状：
- 所有飞书相关操作都失败
- 返回认证错误、token 失效、无权限等信息

处理建议：
1. 检查 `appId` / `appSecret` 是否正确
2. 检查飞书应用是否被重新发布或更换凭证
3. 如你的接入流程依赖重新授权，按实际接入方式重新完成授权

### 5.4 Gateway 未响应

先执行：

```bash
openclaw gateway status
```

如需重启：

```bash
openclaw gateway restart
```

如仍异常，再查看日志文件或启动输出。

## 六、推荐使用示例

### 6.1 发送工作区文件

用户：

```text
把 SOUL.md 发给我
```

期望效果：
- Agent 读取文件
- Agent 通过飞书文件消息把文件发回给用户

用户：

```text

```

期望效果：
- Agent 先生成 `report.md`
- 再把 `report.md` 作为文件发送

用户：

```text
把昨天的日志发我
```

期望效果：
- Agent 找到目标日志
- 使用飞书文件消息发送

## 八、结论

要让飞书文件发送能力正常工作，最关键的是三件事：

1. 飞书开放平台权限齐全且已发布
2. OpenClaw 飞书通道配置正确
3. Gateway 正常运行，并完成了实际发送测试

如果你已经满足以上条件，但仍然只能发文本不能发文件，优先怀疑：
- `im:message.file`
- `im:resource`
- 文件上传链路
